# Git의 작업 영역

- Working directory
  - 파일 생성 및 삭제, 코드 수정
  - 개발자가 작업하는 공간
  - 파일 상태는 아래 두가지로 분류된다.
    1. UnTracked : git이 한번도 관리하지 않은 파일들
    2. Tracked : git이 한번이라도 관리 했던 파일들

- Staging Area
  - 커밋할 변경 사항을 임시로 모아두는 공간
  - 커밋 후보 리스트

- Repository(저장소/ Commit 영역 )
  - 커밋된 변경사항들이 저장되는 Git 데이터베이스
  - 히스토리가 저장되는 공간
  - Commit 단위로 저장, 되돌리기 가능
  - 버전 관리 핵심

# 영역 이동

## Working Directory -> Staging Area

```git
git add [파일명 or 디렉토리]

git add . # 전체 파일 staging area 이동
git add *.json # 현재 디렉토리에 있는 json파일 이동
```

- 그 밖에 옵션
  - `-p` : 변경 내용을 hunk 단위로 staging
  - `-i` : `-p`의 확장판 느낌, 대화형 UI staging
  - `-u` : tracked 된 파일의 수정/삭제만 staging
  - `-A` : 실행 위치와 무관하게 저장소 전체의 변경을 staging
  - `.` : 현재 디렉토리와 하위 디렉토리의 변경만 staging (루트에서 실행 시 `-A`와 동일)

> hunk란?  
> Git에서 hunk는 파일 내에서 연속된 변경 줄의 묶음  
> hunk 단위로 하면 파일 하나에서 연속된 줄 마다 따로 git add 가능

## Staging Area -> Repository

```git
git commit            # vi 모드로 넘어가서 commit 메세지 or 내용 작성 및 저장
git commit -m ""      # "" 사이 메세지 입력 옵션
git commit --amend    # 직전 commit의 메시지나 내용을 수정
```

- `--amend` : staging된 변경 사항을 직전 commit에 합치고, vi로 커밋 메시지를 수정할 수 있음
- `--amend --no-edit` : staging된 변경 사항을 직전 commit에 합치되, 커밋 메시지는 그대로 유지

추가적으로 commit 메세지 작성법은 이 문서를 참고하면 된다.

# 파일 상태 조회

## git status (파일 상태 조회)

```git
git status
```

- **staged 파일, modified 파일, untracked 파일 한번에 확인 가능**
- git add 전에 습관적으로 확인하여 의도하지 않은 파일이 포함되지 않도록 한다
- 주요 표시 상태
  - `Changes to be committed` : staging된 파일 (git add 완료)
  - `Changes not staged for commit` : 수정했지만 아직 staging하지 않은 파일
  - `Untracked files` : Git이 아직 관리하지 않는 새 파일
- 간단히 보기: `git status -s` (short 형식으로 출력)

## git diff (변경점 조회)

```git
git diff              # Working Directory vs Staging Area
git diff --staged     # Staging Area vs 최신 Commit
git diff HEAD         # Working Directory vs 최신 Commit
```

- `git diff` : staging 하기 전, 내가 무엇을 수정했는지 확인
- `git diff --staged` : commit 하기 전, staging된 내용을 최종 확인
- `git diff HEAD` : 마지막 commit 이후 변경된 모든 내용 확인 (staged + unstaged)
- 특정 파일만 비교: `git diff [파일명]`

## git log (Repository 조회)

```git
git log
```

`git commit` 명령어로 작업이 Repository에 저장이 되었다면, `git log`로 커밋 히스토리를 조회할 수 있다.

- 각 커밋의 해시, 작성자, 날짜, 메시지를 확인할 수 있다
- 주요 옵션
  - `--oneline` : 커밋당 한 줄로 간결하게 출력
  - `--graph` : 브랜치 분기/병합을 시각적으로 표시
  - `--all` : 모든 브랜치의 커밋을 표시
  - `--oneline --graph --all` : 조합하면 전체 브랜치 흐름을 한눈에 볼 수 있다
